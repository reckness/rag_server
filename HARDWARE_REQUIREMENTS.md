# RAG Service 硬件设施要求（生产环境）

本文档给出运行本项目所需的硬件建议，分为 **最低可运行**、**推荐生产**、**高并发生产** 三档，并基于以下生产模型组合：

- 主大模型：`Qwen3.5_35B_A3B`
- 辅助模型：`Qwen3.5_9B`
- Embedding：沿用当前线上 embedding 模型（参考现有配置）
- Rerank：两套 `bge` 模型（检索精排 1 套 + 解析阶段语义对比 1 套）

> 适用范围：当前仓库（FastAPI + PostgreSQL + Elasticsearch + MinIO + Embedding/Rerank + PDF 解析）。

---

## 0. 单机一体化生产设计（推荐）

你要求使用 **一台机器（一体机）** 承载完整链路（API、解析、ES、PG、MinIO、模型服务）。

### 0.1 推荐一体机配置（生产可用）

| 硬件项 | 推荐配置 | 说明 |
|---|---|---|
| CPU | **AMD EPYC 9554**（64C/128T, 3.1 GHz）或 **双路 Intel Xeon Gold 6430**（2×32C/128T） | 满足 ES + API + 解析多并发 |
| 内存 | `512 GB` DDR5 ECC（8 x 64 GB） | ES/解析/模型调度均为内存敏感 |
| GPU | **6 × NVIDIA RTX 3090 Ti**（24 GB 每张，共 144 GB） | 见 0.3 节分配方案 |
| 系统盘 | `2 x 3.84 TB NVMe`（RAID1） | 系统 + 日志 |
| 数据盘 | `4 x 7.68 TB NVMe`（RAID10，ES/MinIO 主数据） | 向量检索对 IOPS 敏感 |
| 网络 | 双口 `10/25 Gbps` | 模型调用 + MinIO 上传 |
| 电源 | `2 x 1600W` 冗余（6 张 3090 Ti TDP 约 2100W 总计） | 留 20%+ 余量 |
| 散热 | 机架级散热 / 工作站塔式散热 | 6 张 GPU 散热要求高 |

> **CPU 备选方案**
>
> | 方案 | 型号 | 核心/线程 | 适用场景 |
> |---|---|---|---|
> | 首选（单路） | AMD EPYC 9554 | 64C/128T | 高核密度、单路即可满足 |
> | 次选（单路旗舰） | AMD EPYC 9654 | 96C/192T | 更高并发余量 |
> | Intel 双路 | 2 × Xeon Gold 6430 | 2×32C = 64C/128T | Intel 生态偏好 |
> | Intel 单路工作站 | Xeon w9-3545 | 46C/92T | 工作站形态、单路高核 |
>
> 关键考虑：生产并发需 ES（32 vCPU）+ API/解析（24 vCPU）+ 模型调度（16 vCPU）+ PG/MinIO（12 vCPU）≈ **84 vCPU 常态**，**64 物理核/128 线程** 可覆盖。

### 0.2 资源切分建议（同机隔离）

- `Elasticsearch`：`24~32` vCPU，`192~256 GB` RAM（JVM Heap 建议 `31 GB`）。
- `API + 解析任务`：`16~24` vCPU，`64~96 GB` RAM。
- `PostgreSQL + MinIO`：`8~12` vCPU，`32~64 GB` RAM。
- `模型服务`（LLM/Embedding/Rerank）：GPU 独占部署，CPU 预留 `16+` vCPU 做推理调度。

### 0.3 GPU 分配建议（6 × 3090 Ti, 24 GB/张）

| GPU 编号 | 分配模型 | 显存占用估算 | 部署方式 |
|---|---|---|---|
| GPU-0 / 1 / 2 | `Qwen3.5_35B_A3B`（FP16） | ~70 GB（3 卡 tensor parallel） | vLLM `--tensor-parallel-size 3` |
| GPU-3 | `Qwen3.5_9B`（FP16） | ~18 GB | vLLM 单卡 |
| GPU-4 | Embedding + BGE Rerank（检索精排） | ~4 GB + ~3 GB ≈ 7 GB | 独立进程，共享 1 卡 |
| GPU-5 | BGE Rerank（解析语义对比） | ~3 GB | 独立进程，专用 1 卡 |

> **降配方案（5 × 3090 Ti）**：将 `Qwen3.5_35B_A3B` 改用 **INT8 量化**（~35 GB → 2 卡），总计节省 1 张卡。
>
> **最低方案（4 × 3090 Ti）**：35B 用 INT4 量化（2 卡）+ 9B 与 Embedding 共享 1 卡 + 双 Rerank 共享 1 卡。牺牲推理精度和隔离性。

#### 3090 Ti 部署注意事项

- 3090 Ti 为消费级显卡，**不支持 NVLink**，多卡并行走 PCIe 4.0 x16，卡间带宽约 32 GB/s，对 tensor parallel 有一定性能损失。
- 主板需具备 **6 条 PCIe 4.0 x16 物理槽位**（电气 x8 亦可用，但会进一步损失带宽）。
- 建议选用支持多 GPU 的服务器/工作站主板（如 Supermicro X13/H13 系列或华硕 WRX90 系列）。
- 6 张 3090 Ti 总 TDP 约 **2100W**，需确保电源和散热充足。
- 生产环境若预算允许，建议考虑 **RTX 4090**（24 GB, FP16 性能更强、功耗更优）或 **L40S**（48 GB, 支持 ECC、专为推理设计）。

### 0.4 一体机场景注意事项

- 单机无天然高可用，建议至少做跨机冷备（ES 快照 + PG 备份 + MinIO 异地备份）。
- API 在线流量与离线解析必须分队列限流，避免同机资源争抢。
- 建议设置 CPU/内存/GPU 三层告警，峰值前自动降并发。
- 业务增长后优先拆分：先拆模型服务，其次拆 ES，再拆 API/解析。

## 1. 组件与资源特性

- `RAG API`（FastAPI/Uvicorn）：CPU 中等，内存中等。
- `PDF 解析`（PyMuPDF + 分块 + LLM编排）：CPU/内存波动明显，批处理时峰值高。
- `Elasticsearch`（向量 + BM25）：**内存和磁盘 IO 敏感**，是主要资源消耗点。
- `PostgreSQL`：元数据存储，资源需求较稳。
- `MinIO`：对象存储，主要吃磁盘容量和网络。
- `Embedding/Rerank/LLM`：建议服务化部署；本方案需要同时支持 `Qwen3.5_35B_A3B`、`Qwen3.5_9B` 与双 `bge` rerank 并发调用。

### 1.1 模型角色建议

| 模块 | 建议模型 | 作用 |
|---|---|---|
| 复杂推理/高质量生成 | `Qwen3.5_35B_A3B` | 主回答与复杂任务处理 |
| 轻量任务/解析辅助 | `Qwen3.5_9B` | 目录识别、结构整理、低成本推理 |
| 向量化 | 现有 embedding 模型 | 检索向量生成 |
| 检索精排 | `bge-reranker`（模型A） | RAG 搜索结果 rerank |
| 解析语义对比 | `bge-reranker`（模型B） | 文件解析阶段文本语义比对 |

> 两个 `bge` 模型建议逻辑隔离（独立服务或独立实例），避免解析任务和检索任务互相抢占。

## 2. 最低可运行（开发/测试）

适合：单人调试、少量文档测试（例如几百份以内）。

| 资源 | 建议 |
|---|---|
| CPU | 8 vCPU |
| 内存 | 32 GB |
| 磁盘 | 500 GB NVMe SSD |
| 网络 | 1 Gbps 内网 |
| OS | Linux x86_64（Ubuntu 20.04+/CentOS 7+） |

### 最低可运行说明

- Elasticsearch JVM 建议先给 `8~12 GB` 堆（`-Xms=-Xmx`）。
- 若 Embedding/Rerank/LLM 均使用外部 HTTP 服务，本机可无 GPU。
- 若文档批量导入较大，建议暂停并发任务，避免 ES 和解析进程争抢内存。

---

## 3. 推荐生产（中等规模）

适合：团队共享、持续入库与检索，数据规模约数万到数十万 chunk。

| 节点/服务 | 建议配置 |
|---|---|
| 应用/API 节点（FastAPI） | 16~32 vCPU / 64 GB RAM / 500 GB NVMe |
| 解析任务节点（异步批处理） | 32 vCPU / 64~128 GB RAM / 1 TB NVMe |
| Elasticsearch 节点 | 16~32 vCPU / 64~128 GB RAM / 1~2 TB NVMe |
| PostgreSQL 节点 | 8~16 vCPU / 32~64 GB RAM / 500 GB SSD |
| MinIO 节点 | 8~16 vCPU / 32 GB RAM / 2+ TB SSD/HDD（按文档量扩展） |
| 模型服务节点（LLM + Rerank） | 见第 5 节 GPU 规划 |

### 推荐生产说明

- Elasticsearch 建议独立节点部署，不要和 PDF 批处理混部。
- ES 堆建议设置为机器内存的 `40%~50%`，且不超过约 `31 GB`（常规经验）。
- 磁盘优先 NVMe；向量检索和 bulk 写入对 IO 敏感。
- MinIO 容量按“原始文件 + 中间产物 + 备份”至少预留 `3x`。
- CPU 需预留并发余量：建议应用层常态 CPU 使用率控制在 `60%~70%`，峰值不超过 `85%`。
- 解析任务与在线检索建议分离部署，避免高峰时响应抖动。

---

## 4. 高并发生产（大规模）

适合：高 QPS 检索、持续大批量文档入库。

| 层级 | 建议 |
|---|---|
| API 层 | 3~6 台，单台 32 vCPU / 64~128 GB |
| 解析层 | 3~6 台，单台 32~64 vCPU / 128 GB |
| Elasticsearch | 3 台数据节点起步（副本+高可用），单台 32 vCPU / 128 GB / 2 TB NVMe |
| PostgreSQL | 主从或高可用，16 vCPU / 64 GB |
| MinIO | 分布式部署（4 节点起步） |
| 网络 | 10 Gbps 内网优先 |

### 4.1 多并发 CPU 规划建议

- 在线检索并发与离线解析并发必须隔离队列。
- API 进程建议按 CPU 核数做 worker 规划，并预留系统与 IO 抖动空间。
- 高峰期若出现排队延迟，优先横向扩 API/解析节点，再评估 ES 与模型服务瓶颈。

## 5. GPU 需求（生产模型组合）

### 场景 A：模型走外部 API（推荐）

- 本系统主机：**不强制 GPU**。
- 重点保障 CPU、内存、NVMe、网络连通性。
- 需确保外部模型服务可同时承载：`Qwen3.5_35B_A3B`、`Qwen3.5_9B`、embedding、双 `bge` rerank。

### 场景 B：本地部署（以 3090 Ti 24GB 为例）

| 模型服务 | 精度 | 显存估算 | 3090 Ti 卡数 |
|---|---|---|---|
| `Qwen3.5_35B_A3B` | FP16 | ~70 GB | 3 张（tensor parallel） |
| `Qwen3.5_35B_A3B` | INT8 | ~35 GB | 2 张 |
| `Qwen3.5_35B_A3B` | INT4 | ~18 GB | 1 张（牺牲精度） |
| `Qwen3.5_9B` | FP16 | ~18 GB | 1 张 |
| Embedding（现有模型） | FP16 | ~2-4 GB | 可与 9B 或 Rerank 共享 |
| Rerank-检索（bge） | FP16 | ~2-3 GB | 1 张（可与 Embedding 共享） |
| Rerank-解析语义对比（bge） | FP16 | ~2-3 GB | 1 张（建议独占避免争抢） |

**推荐总计：6 张 3090 Ti**（FP16 全精度、完全隔离）
**经济方案：5 张**（35B 用 INT8）
**最低方案：4 张**（35B 用 INT4 + 部分模型共享）

> 建议将两套 `bge` rerank 分别部署在独立实例，保证检索链路时延稳定。
>
> 如使用远端模型服务，请重点确认网络延迟、带宽和限流策略。
>
> **GPU 升级路径**：若预算充足，推荐 **4 × RTX 4090**（24 GB, 性能更强）或 **2 × L40S**（48 GB, 支持 ECC）+ **2 × RTX 4090** 替代 6 张 3090 Ti。

## 6. 容量估算建议（非常重要）

粗估方法（用于预留 ES + MinIO）：

1. 统计日均新增文档数 `D`。
2. 估算单文档平均 chunk 数 `C`。
3. 估算单 chunk 在 ES 中平均占用（含向量、倒排、元数据）`S`（可先按 `3~10 KB` 粗估，实际需压测校正）。
4. 年增长量约：`D * C * S * 365`。
5. 再乘以副本、索引开销、预留空间系数（建议 `2~3x`）。

## 7. 基础设施 Checklist（上线前）

- [ ] ES、PG、MinIO 均已独立监控（CPU/内存/磁盘/IOPS/延迟）。
- [ ] ES 磁盘使用率告警（70%/85% 两级）。
- [ ] PG 定期备份 + 恢复演练。
- [ ] MinIO 生命周期策略与归档策略。
- [ ] API 与模型服务连通性压测（embedding/rerank/LLM）。
- [ ] 文档批处理并发参数已按硬件调优。

---

## 8. 一句话建议

- **开发测试**：`8C/32G/500G NVMe` 可跑通。
- **稳定生产**：至少将 **Elasticsearch 独立到 64G+ 内存节点**，并把解析与在线检索分层部署。
- **大规模**：优先扩 ES 与模型服务，其次扩 API/解析 CPU 节点，确保多并发下可用性。
